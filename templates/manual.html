User Manual & API Reference  ·  v2.12.94  ·  2026-03-16
AI-augmented voice agent for Icom amateur radio transceivers

01 Introduction

RigGPT is a Flask-based web application that bridges AI text-to-speech engines with Icom amateur radio transceivers via the CI-V serial protocol. It enables automated voice transmissions, AI-generated debates (Acid Trip mode), SSTV image transmission, beacon scheduling, transmission logging, and full Discord integration.

The interface is a dense, terminal-style operator console designed for speed and efficiency. All primary functions are keyboard-accessible. The application runs as a systemd service on Debian x86_64 and serves a single-page UI at port 5000.

Target Platform
Debian 13 (trixie) amd64, Python 3.13
Tested with Icom IC-7610 primary
Server Stack
Flask + gunicorn + gevent
SQLite logging · APScheduler beacons

02 Quick Start

  1. Extract the zip to any directory.
  2. Run sudo bash INSTALL.sh — creates the riggpt service user, installs dependencies, configures systemd, and starts the service.
  3. Open http://<host-ip>:5000 in a browser.
  4. Go to Config → Connection, select your radio model, set the serial port, and click CONNECT.
  5. Go to Config → Integrations, add API keys for whichever TTS/AI engines you want to use.
  6. On the Transmit tab, type a message, select an engine/voice, and click TRANSMIT.
Amateur Radio Licence Required
RigGPT controls PTT and transmits on amateur radio frequencies. You must hold a valid amateur radio licence and operate within your licence class privileges and local regulations.

04b AI Prompt Architecture

RigGPT does not use Ollama modelfiles. All AI personality is injected at runtime via the role: 'system' message in the Ollama /api/chat request. This means persona and canon can be changed mid-session without reloading models.

Injection Points

There are four independent prompt pipelines, each assembling a system message differently:

FeatureSystem Prompt AssemblySource
AI Tab (Generative Chat) ai_persona + optional Qdrant memory context PERSONA textarea on AI tab, persisted in app_settings.json
Acid Trip (Agent Debates) canon + persona + debate topic + oracle instruction + discord influence + output guardrail Per-agent textareas on Acid Trip tab, or loaded from /home/riggpt/canon/a/ and /home/riggpt/canon/b/
SENTIENT (Whisper Listener) sentient_persona + source-aware context ("You just heard a human operator / another AI radio station say...") PERSONA textarea on SENTIENT tab
Alexa Mode ALEXA_PERSONA (hardcoded constant) Python constant in app.py

Canon vs Persona

Canon is character lore — background knowledge that shapes the agent's worldview. It is injected first in the system prompt so the model absorbs it as foundational context. Canon is loaded from .md files on the server or typed directly into the Canon textarea.

Persona is behavioral instruction — how the agent should talk, its tone, personality, and constraints (e.g. "Keep responses under 25 words. Speak like a tired pirate."). It follows the canon in the system prompt.

Together they form the complete system message:

[Canon text — character background, lore, knowledge]

[Persona text — behavioral rules, tone, constraints]
Debate topic: [topic].
Reply in 1-3 sentences. Stay in character.
[Oracle instruction if enabled]
[Discord audience influence if active]

OUTPUT: Only your spoken words. No labels, no instructions, no metadata.

Persistence

All persona and canon values are saved to app_settings.json via the settings flush mechanism. They survive restarts, page reloads, and re-deployments. Canon files on disk (/home/riggpt/canon/) are never touched by the installer.

Memory Augmentation (Qdrant)

When Community Memory is enabled, the AI Tab chat pipeline appends Qdrant context to the persona. This includes relevant Discord community messages, user profiles, and conversation threads retrieved via semantic search. Acid Trip agents receive a pre-trip briefing (richer, one-time context) at turn 0, then lighter per-turn context thereafter. See the Community Memory section for details.

SDR waterfalls use logarithmic color maps. RigGPT applies sqrt() amplitude boost and per-row normalization automatically. Default contrast=2.0 + equalize=on for maximum visibility. Images are auto-flipped vertically so they appear right-side-up on scrolling waterfalls.

13 Acid Trip Tab

Acid Trip is a real-time AI debate engine. Two AI agents engage in an open-ended conversation, their voices transmitted live over the radio. Discord chat drives the conversation in real time.

Overview

Each "trip" runs for a configured number of turns. Each half-turn, the active agent generates a response using its persona, canon, and the conversation history, synthesises it via its assigned TTS engine, and transmits. Discord messages are woven into the generation prompt as a live audience signal.

Design Philosophy
Agents are intentionally nameless. They know each other exists through the conversation but have no handle to address each other directly. This creates a more atmospheric, less chatty exchange. The output guardrail in every system prompt explicitly instructs the model not to use names.

Discord Integration

When Discord is configured and connected, Acid Trip continuously reads the Discord message buffer and uses it to influence agent generation.

Discord Influence Slider (1–10)

Controls how strongly Discord messages are framed in the system prompt:

LevelLabelEffect
1–3Minimal / Low / NoticeableSoft background suggestion — "somewhere at the back of your mind..."
4–5Moderate / PresentAudience awareness — Discord messages may surface naturally.

How Influence Actually Works
The system context is rebuilt from scratch on every turn, so adjusting the slider mid-debate takes effect immediately on the next generation. The slider at 1–3 uses soft prose framing; at 4+ it activates the Lookback mechanism.

Discord Lookback

The primary mechanism for Discord to drive the conversation. When Influence is 4 or above and Lookback N > 0, the Lookback system fires on qualifying turns and replaces the final user message in the generation prompt with a synthesised Discord-driven directive. The agent responds directly to what Discord said, not to the prior agent turn.

ControlDescription
DISCORD LOOKBACK NHow many recent Discord messages to pull in (0 = off, max 10).
DRIVE EVERY N TURNSFire frequency. 1 = every turn, 2 = every other turn, etc.

When Lookback fires, a blue 📡 DISCORD DRIVE [N msg]: ... banner appears in the chat transcript showing exactly what content was used as the prompt. The status field shows the last turn it fired on.

The prompt framing scales with Influence level:

  • 4–5: "Your listeners have been talking: ... Let what they said color what you say next."
  • 6–7: "Your audience just said: ... React to this directly. Make it the springboard."
  • 8–9: "STOP. Your audience is talking and you cannot ignore it: ... Let it hijack the conversation."
  • 10: "EVERYTHING STOPS. ... Forget the debate."
Chaos Queue Priority
If a chaos queue event (SPICY button, scheduled event, control channel inject) arrives on the same turn as a Lookback fire, the chaos event takes priority. Lookback only fires on turns where the chaos queue is empty.

Discord Interact Mode

Discord Interact is a reactive mode where Discord messages trigger agent responses directly, rather than running on a fixed turn schedule.

Two Operating Modes

ConditionButton LabelBehaviour
No trip running📡 INTERACT: LISTENINGWaits for Discord activity. When a new message arrives, launches a fresh tripStart() with the Discord context as the seed. START/STOP are locked while listening.
Trip already running📡 INTERACT: MID-TRIPArms without interrupting the current debate. Incoming Discord messages are pushed directly into the inject and chaos queues. The running agents pick them up on their next turn. START/STOP remain usable.

When a mid-trip Interact-armed debate ends naturally, the button automatically transitions to LISTENING mode and waits for the next Discord trigger to start a fresh trip. Pressing STOP manually disarms Interact entirely.

Chaos / Wildcard

Chaos events are plain-text interrupts injected into the agent's next generation as the active user message, overriding the normal conversational flow for that turn.

Sources

  • SPICY button — manual inject of a random event from the built-in chaos list
  • WILDCARD auto-chaos — fires every N turns automatically (toggle + frequency control)
  • Scheduled chaos — per-turn events defined in Mode Presets or the scenario schedule editor
  • Control channel — !inject [text] command via Discord
  • Discord Lookback / Interact — Discord content treated as chaos when it fires

16 SENTIENT Tab

The radio becomes aware. SENTIENT listens to incoming audio from the IC-7610's USB audio capture, transcribes speech using OpenAI Whisper (or faster-whisper), generates AI responses via Ollama, and transmits them back over the air. Keyboard shortcut: @.

    → Ollama AI generation (source-aware persona)

Whisper Engines

RigGPT tries faster-whisper (CTranslate2, 3-4x faster on CPU) first, falls back to openai-whisper. Install one:

pip3 install faster-whisper --break-system-packages   # recommended
pip3 install openai-whisper --break-system-packages   # fallback

The base model (74MB) is recommended for CPU — transcribes 10 seconds of audio in 1-2 seconds with faster-whisper. The tiny model is faster but less accurate. Models larger than small are too slow without a GPU.

Bot-to-Bot Handshake

Every bot transmission is preceded by a 150ms dual-tone burst: 1337 Hz + 2600 Hz simultaneously. When another RigGPT instance hears this, FFT analysis detects both frequency peaks standing 8x above their local spectral neighborhood, and the source is tagged as BOT. The AI persona adapts: "You just heard another AI radio station say..."

Put one RigGPT on a frequency in your shack, another at a friend's QTH, both in HIVE MIND mode. They detect each other's handshake tones and carry on autonomous AI conversations across the ionosphere.

HF Audio Considerations

HF radio audio has a very narrow signal-to-noise ratio — speech may only be 1.3-2x above the noise floor (vs 10x+ for microphone audio). The energy threshold should be set using the 🎯 CALIBRATE button, which measures the noise floor and sets the threshold to 1.3x above it. The 🎤 TEST CAPTURE button records 2 seconds and reports audio statistics.

A hallucination filter discards common Whisper artifacts on noise ("Thank you for watching", "Subscribe", etc.).

40 Community Memory

The Community Memory system ingests your Discord channel history into a local Qdrant vector database, builds per-user personality profiles using a local LLM, and injects relevant community context into Acid Trip agent system prompts at runtime. Agents develop historical awareness of their audience — running jokes, technical obsessions, who argues about what — grounded in real conversation history.

RAG, Not Training
Model weights are never modified. Community knowledge is injected as plain prose into the system prompt at runtime. Swap models freely — the memory system works with any engine. Knowledge updates take effect on the next ingest run with no restart required.

How Qdrant Works

Qdrant is a vector database — it stores documents alongside their mathematical representations (vectors) and answers queries like "find me the 5 documents most semantically similar to this phrase" in milliseconds. RigGPT uses it as a long-term memory layer that no SQL database could replicate: meaning-based retrieval rather than keyword matching.

Vectors and Embeddings

Every piece of text stored in Qdrant is first passed through nomic-embed-text — a local Ollama embedding model that converts text into a 768-dimensional vector (a list of 768 floating-point numbers). Semantically similar texts produce geometrically close vectors. "Antenna tuner SWR problems" and "my coax has high VSWR" will have closer vectors than "antenna tuner SWR problems" and "I had pizza for lunch." Qdrant searches using cosine similarity: the angle between vectors, where 0° = identical meaning and 90° = unrelated.

The Four Collections

CollectionOne document = Payload fieldsRebuilt
discord_messages One Discord message content, author_username, author_display, author_id, author_hash, timestamp, has_attachment New messages only (delta)
user_profiles One community member's behavioral profile display_name, communication_style, dominant_topics, humor_style, audience_role, recurring_themes, emotional_register, technical_depth, message_count Only users with new messages (delta)
conversation_threads One clustered conversation thread summary, participant_displays, participant_usernames, message_count, participant_count, timestamp, first_message_id New delta messages at 100% sample; full ingest at 40% sample + all 5+ message threads
channel_themes Single channel-wide summary document community_character, running_jokes_or_references, distinctive_phrases, recurring_topics, emotional_register, notable_events Every delta run (fast — samples recent messages only)

Thread Clustering Algorithm

The cluster_threads() function groups messages into conversation threads without requiring Discord thread IDs. A new thread starts when both of these are true: the message does not reference (reply to) any message in the current thread, and more than 5 minutes have elapsed since the last message. This catches natural conversation breaks — topic shifts after silence — as well as explicit reply chains. Only threads with 2 or more participants are stored.

Thread summaries are generated by qwen3:14b with a prompt that names participants and captures what was actually said — not a generic summary, but a specific one: "Dave and K4ABC argued about whether FT8 counts as a real contact, with K4ABC eventually conceding the point."

Point IDs

Qdrant requires integer point IDs. Discord message IDs are snowflakes — 64-bit integers encoding timestamp + worker + sequence. RigGPT stores the lower 63 bits (int(message_id) & 0x7FFFFFFFFFFFFFFF) to stay within Qdrant's signed 64-bit range. This makes every message's point ID stable and deterministic — re-running ingest on the same message is an upsert, not a duplicate insert.

Payload Indexes

Qdrant payload indexes allow fast filtering without scanning all vectors. RigGPT creates indexes on author_hash, author_username, and timestamp_unix in discord_messages, and on participant_usernames and timestamp_unix in conversation_threads. These are used by the ingest pipeline to efficiently retrieve all messages for a given user when rebuilding their profile.

Context Injection

When memory is enabled, memory.build_context() or memory.build_briefing() is called for every /api/ai/chat request from Acid Trip. The result is appended to the system prompt as plain prose — no headers, no JSON, indistinguishable from the core persona text.

Turn 0: Pre-Trip Briefing

On the very first turn of a trip (trip_turn=0), both agents receive a richer one-time briefing built by build_briefing(). This gives agents shared grounding before a single word is said on air:

  1. Past thread summaries — the top 4 conversation threads from Qdrant most relevant to the trip topic. Named participants included. "Dave and K4ABC argued about this for three days in January."
  2. Relevant audience members — the 4 user profiles most closely matched to the topic, with name, role, usual topics, and humor style.
  3. Running community references — pulled from channel_themes.running_jokes_or_references. Phrases and in-jokes the agents can drop naturally.

Turns 1–3: Per-Turn Context

Built by build_context() with three layers:

  1. Community DNA — the channel-level character profile, cached for 1 hour. Always included. Covers overall vibe, humor culture, recurring phrases, what this community actually cares about.
  2. Audience profiles — 2–3 user profiles most semantically relevant to the current topic, retrieved fresh each turn. Agents A and B use different query angles so they encounter different community facets.
  3. Thread summaries — 3 conversation threads close to the topic. Injected as "Past conversations in this community on related topics."
  4. Past message echoes — a semantic search for raw Discord messages close to the topic. Injected as background memory: "this community has talked about similar things before." Turns 0–3 only.

Turns 4+: Lean Mode

Community DNA and audience profiles only. Past echoes and threads are dropped to keep the prompt lean. The Discord Lookback system carries live audience signal from this point forward.

What Agents Actually Receive
Memory context is injected as natural prose, not labeled blocks. The agent sees something like: "The people listening to you right now include: Dave (technical expert) — usually talks about weak signal modes, humor style: dry and literal. [K4ABC] argues regularly with Dave about FT8 operating ethics. Past conversations on related topics: [Dave, K4ABC] Dave made the case that FT8 contacts don't count as real QSOs — K4ABC eventually conceded after a long argument. Running references you can drop naturally: 'the SWR incident', 'Dave's shack fire of 2021'."

43 Troubleshooting

No audio output / PTT activates but no sound

  • Check the ALSA device in Config → Audio Devices. The IC-7610 appears as a Burr-Brown / PCM2901 USB audio device.
  • Verify the radio is in USB mode and the USB audio codec is selected in the radio's audio settings (Menu → Connectors).
  • Run aplay -l on the server to list available cards. Update RIGGPT_AUDIO_DEVICE in the systemd unit if needed.
  • Check journalctl -u riggpt -f for sounddevice errors. "Invalid device" means the card index changed — re-list with aplay -l after reconnecting USB.

PTT not working / radio not keying

  • Verify CI-V address. Use Config → Debug → CIV PROBE to scan for the responding address.
  • Check baud rate matches the radio's CI-V baud setting (Menu → Connectors → CI-V Baud Rate on the IC-7610).
  • Ensure the riggpt service user is in the dialout group: groups riggpt. If not: usermod -aG dialout riggpt then restart the service.
  • Check journalctl -u riggpt -f for serial port errors. "Permission denied" on /dev/ttyUSB0 is the dialout group issue.

Transmission cuts off mid-sentence

  • The WAV file may be longer than the PTT timeout. Increase the TX timeout value in Config → Radio or use a shorter MAX TOK value for AI generation.
  • For Acid Trip: check the DELAY setting. Insufficient gap between turns can cause the next PTT to pre-empt the current transmission.

AI generation fails / "AI ERROR" in Acid Trip log

  • For Ollama: verify the host is reachable (curl http://your-host:11434) and the model is loaded (ollama list). The model must appear in the list — if it was recently pulled, wait for the pull to complete.
  • For OpenAI / Gemini / ElevenLabs: check the API key in Config → Integrations. Use the TEST button. A 401 error means wrong or expired key. A 429 means rate limit — reduce request frequency or upgrade tier.
  • Acid Trip debug log (bottom of Acid Trip tab, DEBUG section) shows full error details including HTTP status codes and response bodies.
  • "latin-1 encoding error" in server logs means a non-ASCII character (em-dash, smart quote) slipped into a response string. This is a backend bug — report it with the offending response text.

TTS produces no audio / silent WAV file

  • Speechify: exit code 69 means the Speechify desktop app is not running or is not logged in. Launch Speechify on the server machine and ensure you are logged in.
  • Edge TTS / gTTS: requires outbound internet. Check connectivity from the server.
  • Piper / eSpeak: verify the model file path is correct in the TTS engine config. Run piper --help to confirm the binary is in PATH.
  • FastKoko: verify the Kokoro service is running at the configured URL. Check curl http://localhost:8880/health.
  • For any engine: check /opt/riggpt/wav_cache/ — if WAV files are being created but are 0 bytes or very small, the TTS call succeeded but rendered silence. This is usually a voice/model mismatch.

Agent speaks its system prompt aloud (prompt leakage)

  • The backend _sanitise_trip_reply() strips known leakage patterns. Check the DEBUG log to see what was stripped.
  • Weaker or smaller models (under 7B parameters) are most prone to this. Try a larger model.
  • Reduce MAX TOK — shorter forced output leaves less room for the model to ramble into its own instructions.
  • Temperature above 1.4 dramatically increases leakage probability. Reduce TEMP to 0.9–1.2.
  • If the persona text is very long (heavy Canon loading), some models will echo it. Trim the canon or use a model with stronger instruction-following.

Discord not receiving messages

  • Verify the bot token, guild ID, and channel ID in Config → Discord. Click APPLY after any changes.
  • The Discord poller runs every 8 seconds. Messages should appear in the Acid Trip DEBUG → 📡 DISCORD tab within ~10 seconds of being posted.
  • Ensure the bot has Read Message History and View Channel permissions on the configured channel.
  • If the bot was recently invited or permissions changed, it may need to be kicked and re-invited to pick up new permission scopes.
  • Check journalctl -u riggpt -f for 401 errors from the Discord poller — these indicate an expired or revoked bot token.

Discord Interact does nothing when messages are posted

  • The INTERACT button must be active (cyan) before or during a trip. It is not retroactive.
  • During a running trip, Interact pushes to the inject queue — the agent will pick it up on the next turn, not immediately.
  • If INFLUENCE is set to 0, Discord messages are received but have no weight in the system prompt. Raise it (default: 7).
  • The control channel is excluded from Interact by design — only audience channel messages drive agent behavior.

Acid Trip START button does nothing

  • Open browser developer console (F12 → Console). Any JavaScript errors will appear there — these are the definitive diagnosis.
  • Ensure at least one AI provider is configured with a model selected. The model dropdown must not be empty.
  • If Discord Interact is armed (button is cyan), START is locked. Disarm Interact first, then START.
  • Hard-reload the page (Ctrl+Shift+R) to clear any stale cached JavaScript — especially after a version upgrade.

Trip runs but agents give identical or very similar responses

  • Verify Agent A and Agent B have distinct personas. If both fields are empty or identical, the agents have no differentiation.
  • Set TEMP to at least 0.9. Very low temperature causes deterministic, repetitive output.
  • Ensure Agent A and Agent B are using different voice/TTS configurations — identical voices make the problem harder to notice on air.
  • Check that SEED is blank or different for each agent. A fixed identical seed forces identical sampling.

Agent response gets cut off / ends mid-sentence

  • MAX TOK is set too low. The response hit the token limit before completing. Raise MAX TOK or shorten the persona prompt to leave more room for the actual response.
  • For models with a small context window, very long conversation histories (many turns) push early content out of context. The model loses track and truncates. Reduce TURNS or switch to a larger context model.

Memory STATUS shows 0 messages / collections not found

  • Qdrant is not reachable. Verify the host address in Config → Community Memory and run curl http://your-qdrant-host:6333/healthz from the RigGPT server.
  • The full ingest has not been run yet. Run python3 /opt/riggpt-memory/riggpt-ingest.py --full.
  • If collections were dropped and recreated but the ingest shows "0 messages fetched," the checkpoint file is stale. Delete it: rm /home/riggpt/memory_checkpoint.json and re-run --full.

Delta ingest fails with 401 Unauthorized

  • The credentials wrapper at /opt/riggpt-memory/run-delta.sh still contains the placeholder token. Edit it: nano /opt/riggpt-memory/run-delta.sh and replace REPLACE_WITH_YOUR_BOT_TOKEN with the actual bot token.
  • The bot token must be prefixed with Bot (capital B, space) — e.g. Bot MTIzNDU2Nzg5.XXXXXX.
  • If the token was recently regenerated in the Discord Developer Portal, update the credentials wrapper and the RigGPT Config → Discord panel.

Profile generation produces low-quality or empty profiles

  • The profile LLM (qwen3:14b by default) must be running on Ollama. Verify with docker exec ollama ollama list.
  • If profiles are generating but fields are empty or nonsensical, the model may be emitting thinking-mode <think> blocks before JSON. This is handled automatically in v2.12.18+. Ensure you are running the latest ingest script.
  • Users with fewer than 10 messages are skipped during profiling. This is intentional — insufficient data produces unreliable profiles.
  • Run --profiles-only after confirming the LLM is healthy to rebuild without re-ingesting messages.

Ollama GPU access lost after host reboot

  • docker exec ollama nvidia-smi returns "Failed to initialize NVML: Unknown Error" — the container started before the NVIDIA kernel module was ready.
  • Fix immediately: docker restart ollama then verify with docker exec ollama nvidia-smi.
  • Permanent fix — make Docker wait for the NVIDIA runtime on every boot:
mkdir -p /etc/systemd/system/docker.service.d
echo -e "[Unit]\nAfter=nvidia-persistenced.service" \
  > /etc/systemd/system/docker.service.d/nvidia-wait.conf
systemctl daemon-reload

Memory context not appearing in agent responses

  • Confirm memory is enabled in Config → Community Memory (checkbox must be checked and saved).
  • Click STATUS — if it shows 0 profiles or 0 messages, the ingest has not run or Qdrant is unreachable.
  • Check the Acid Trip DEBUG log for "Memory inject failed" or "Memory context injected (0 chars)" lines. The former indicates a Qdrant connection error. The latter means the semantic search returned no results above the similarity threshold — the topic may be too niche relative to what was actually discussed in Discord.
  • Use TEST QUERY in the Config panel to manually verify retrieval for your trip topic.

RigGPT service won't start / crashes on startup

  • Check journalctl -u riggpt -n 50 --no-pager for the traceback. Common causes:
  • Port 5000 already in use: another process has the port. ss -tlnp | grep 5000 to identify it.
  • Database migration error: the SQLite schema changed between versions. Stop the service, run the migration manually, restart.
  • Missing Python dependency: ModuleNotFoundError in the log. Install with pip3 install <module> --break-system-packages as the riggpt user.
  • Config file permission error: /home/riggpt/app_settings.json must be owned by the riggpt user. chown riggpt:riggpt /home/riggpt/app_settings.json.

Page loads blank / all tabs empty after upgrade

  • Hard-reload with Ctrl+Shift+R (or Cmd+Shift+R on Mac) to clear browser cache. Stale JS from a previous version is the most common cause.
  • The server sends cache-busting headers but some browsers ignore them for the main document. If hard-reload doesn't fix it, open DevTools → Application → Clear Storage → Clear site data.
  • Check browser console (F12) for JavaScript errors — a TDZ (Temporal Dead Zone) or reference error will prevent the entire page from initializing.

Export/Import not saving all settings

  • Click EXPORT only — do not navigate away or switch tabs before the download completes. The export snapshot captures DOM state at click time.
  • If Agent A/B configuration is missing after import, the relevant keys may have been added after your export was created. Check the version number in the export JSON — settings keys change between versions and older exports may be missing new fields.
  • The API keys export is optional and separate from the main settings export. If you need to migrate API keys to a new machine, enable "include API keys (plaintext)" in the export dialog.
